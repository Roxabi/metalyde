## Issue Lifecycle

```
Create → Triage → Prioritize → Assign → Implement → Review → Close
```

---

## Issue Metadata — Labels-First Model

Issue metadata lives on **GitHub labels + native issue fields** — there is no required project board. All label families are already bootstrapped in the repo.

### Label families

| Family | Labels | Purpose |
|--------|--------|---------|
| **Priority** | `P0-critical` `P1-high` `P2-medium` `P3-low` | Urgency |
| **Size** | `size:S` `size:F-lite` `size:F-full` | Dev tier / effort |
| **Status** | `status:Backlog` `status:Analysis` `status:Specs` `status:In Progress` `status:Review` `status:Done` | Seeded on create by `/issue-triage`; not written on `set` |
| **Lane** | `graph:lane/<lane>` | Dep-graph swim-lanes (a1 a2 a3 b c1 … standalone) |
| **Blocked** | `blocked` | Issue is waiting on another issue |

### Native GitHub fields

| Field | Storage |
|-------|---------|
| **Issue Type** | Org-level native types (`feat fix refactor docs test chore ci perf epic research`) — inherited per repo |
| **Milestone** | Native GitHub milestone (per-repo) |
| **Sub-issues** | Native GitHub sub-issue relations (REST `…/sub_issues`, `…/parent`) |
| **Blocked-by / Blocking** | Native GitHub dependency relations (`…/dependencies/*`) |

> Status is tracked primarily via the **native open/closed state**. `/issue-triage set` does not write `status:*` labels; only `/issue-triage create` seeds one on creation.

## Creating Issues

### From Templates

Use the provided GitHub templates:

- **Bug Report**: For reporting bugs with reproduction steps
- **Feature Request**: For proposing new features

### From the CLI

Use `/issue-triage create` to create an issue with all fields set in one shot:

```bash
/issue-triage create \
  --title "feat: notification preferences" \
  --body "Allow users to configure notification channels" \
  --label "feature" \
  --size M --priority High \
  --status Backlog \
  --parent 163 \
  --blocked-by 42
```

### Minimum Requirements

Every issue should have:

- Clear, actionable title
- Description with context and acceptance criteria
- Related issues linked (if any)

## Triage

Triage ensures every issue has a **Size** and **Priority** assigned.

### Size

| Size | Label | Description |
|------|-------|-------------|
| **S** | `size:S` | Small — ≤3 files, single domain, no unknowns |
| **F-lite** | `size:F-lite` | Feature lite — clear scope, 4–10 files |
| **F-full** | `size:F-full` | Feature full — multi-domain, new arch, or unknowns |

### Priority

| Priority | Label | Description | Action |
|----------|-------|-------------|--------|
| **Critical** | `P0-critical` | Blocking or critical | Do immediately |
| **High** | `P1-high` | Important for current milestone | Do this sprint |
| **Medium** | `P2-medium` | Should be done soon | Plan for next sprint |
| **Low** | `P3-low` | Nice to have | Backlog |

### Triage Process

Use the `/issue-triage` skill (provided by `dev-core` plugin) to list untriaged issues, set fields, create new issues, and manage parent/child relationships:

```bash
# List issues missing Size or Priority
/issue-triage list

# Set size, priority, and status
/issue-triage set 42 --size M --priority High
/issue-triage set 42 --status "In Progress"

# Create a new issue with full setup
/issue-triage create \
  --title "feat: avatar upload" \
  --size M --priority High --parent 163

# Manage parent/child (sub-issue) relationships
/issue-triage set 164 --parent 163
/issue-triage set 163 --add-child 164,165
/issue-triage set 164 --rm-parent

# Manage blocked-by dependencies
/issue-triage set 91 --blocked-by 117
/issue-triage set 91 --rm-blocked-by 117
```

## Issue Lifecycle Automation

There is no project board. Lifecycle is driven by native GitHub state:

- **Open** = active (Backlog → Analysis → Specs → In Progress → Review)
- **Closed** = done or won't-do

PR→issue closing is handled by closing keywords in PR descriptions (`Closes #N`). The `auto-merge` workflow merges on green and the closing keyword automatically closes the linked issue on merge.

## Blockers & Dependencies

### Adding a Blocker

When an issue blocks another, use the `/issue-triage` skill (preferred) or the raw GraphQL API:

```bash
# Preferred — via /issue-triage
/issue-triage set 51 --blocked-by 19
/issue-triage set 19 --blocks 51
```

If using the API directly:

1. Use GitHub's native `blockedBy` dependency (source of truth):
   ```bash
   # Make #51 blocked by #19
   gh api graphql \
     -F query='mutation($issueId: ID!, $blockingId: ID!) {
       addBlockedBy(input: { issueId: $issueId, blockingIssueId: $blockingId }) {
         issue { number }
         blockingIssue { number }
       }
     }' \
     -f issueId="$(gh api repos/OWNER/REPO/issues/51 --jq '.node_id')" \
     -f blockingId="$(gh api repos/OWNER/REPO/issues/19 --jq '.node_id')"
   ```

2. Add the `blocked` label to the blocked issue:
   ```bash
   gh issue edit <number> --add-label "blocked"
   ```

3. Reference the blocked issue in the blocker's body (e.g., "Blocker for #19")

### Resolving a Blocker

When a blocking issue is closed:

1. Remove the `blocked` label from previously blocked issues:
   ```bash
   gh issue edit <number> --remove-label "blocked"
   ```

2. Verify unblocked issues are ready to work on

### Viewing Dependencies

Use the `/issues` skill (provided by `dev-core` plugin) to see the full dependency graph:

```bash
/issues
```

The output shows block status for each issue:

| Icon | Meaning |
|------|---------|
| `✅` | Ready — no open blockers |
| `⛔` | Blocked — waiting on other issues |
| `🔓` | Blocking — other issues depend on this |

## Best Practices

- **One issue per concern**: Don't bundle unrelated changes
- **Link related issues**: Use GitHub references (`#42`) and `blockedBy`
- **Close with PR**: Reference the issue in your PR (`Closes #42`)
- **Update status**: Close issues when done; use `status:*` labels only if finer-grained tracking is needed
- **Break down F-full**: Issues sized `size:F-full` should be broken into smaller sub-issues where possible
